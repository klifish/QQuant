# 从零打造个人交易系统：90天实践计划（修订版）

> 目标：把炒股从"凭感觉操作"转变为一个可验证、可复盘、可迭代的系统工程。  
> 适用方向：A股日线级别趋势波段交易。  
> 重要提醒：本文仅用于学习、研究和个人系统设计，不构成投资建议。

---

## 0. 核心原则

你不是在开发"预测软件"，而是在开发：

> 一个在不确定市场中控制风险、重复执行、持续复盘的决策系统。

系统优先级：

```text
风险控制 > 数据质量 > 执行纪律 > 回测可靠性 > 策略收益 > 自动化程度
```

自动化只是放大器。规则正确，它放大纪律；规则错误，它放大亏损。

正确路线：

```text
交易规则文档 → 信号系统 → 回测验证 → 模拟交易 → 小资金实盘 → 半自动执行
```

---

## 0.1 时间线预期管理

本计划90天目标是完成一个**可用的信号系统 + 经过初步验证的回测框架**。

| 阶段 | 时间 | 产出 |
|---|---|---|
| 90天 | 本计划 | MVP信号系统 + 回测框架 + 开始模拟交易 |
| 120-150天 | 延伸阶段 | 充分模拟验证 + 小资金实盘 |

不要期望90天结束就能放真钱。数据处理、回测框架和模拟盘都比预期耗时，这是正常的。

## 1. 第一阶段目标

第一版系统不应该试图覆盖全市场、预测涨停、抓龙头或做高频交易。

合理起点：

> 在A股中寻找中短期趋势向上的股票，控制仓位，持有几天到几周，跌破规则就退出。

---

## 2. 系统边界定义

写代码前，先写"交易系统说明书"。边界不清，回测和复盘都会失真。

### 2.1 交易市场

第一版只做：

- A股
- 日线级别
- 不做ST
- 不做退市风险股
- 不做新股（上市不足一年）
- 不做长期停牌股票
- 不做近20日平均成交额过低的股票（建议阈值：5000万元以上）
- 不做一字涨停和一字跌停股票

### 2.2 交易周期

日线级别波段交易，而不是分钟级、日内、高频或打板。

原因：
- 噪音较少
- 交易频率较低
- 适合非职业交易者
- 容易用Tushare等数据源回测
- 更适合建立稳定规则

### 2.3 第一阶段目标

不要把第一阶段目标设为"年化收益多少"，合理目标是：

- 控制最大回撤
- 形成稳定交易规则
- 验证系统是否有正期望
- 训练自己执行纪律
- 识别策略在哪些市场环境下失效

第一版系统的任务是证明：

> 这套规则在历史数据、模拟交易和小资金实盘中不会明显失控。

## 3. 数据层：提前规划

**在开始写代码前，先确认数据成本。** 这是最容易被忽视的前置条件。

### 3.1 积分预算

| 数据类型 | 所需积分级别 | 说明 |
|---|---|---|
| 日线行情（全量历史） | 120积分 | 5年以上历史需要足够积分 |
| 停牌数据 | 2000积分 | 关键数据，必须有 |
| 财务数据 | 2000积分 | 第一版可以暂不用 |
| 指数成分 | 120积分 | 用于大盘环境判断 |

**行动项（第1周完成）**：
1. 注册Tushare Pro，查看当前积分
2. 跑一个数据验证脚本，确认以下数据可正常下载：
   - 全量A股日线（至少2015年至今）
   - 停牌信息
   - ST信息
   - 指数日线（沪深300、中证500）

如果停牌数据积分不够，提前充值或找替代方案（AKShare、baostock），不要到第3周才发现数据缺口。

### 3.2 数据下载注意事项

- **频率限制**：Tushare API有每分钟调用限制，全量历史下载需要分批，预计需要数小时
- **复权**：使用前复权（qfq），所有历史价格统一复权后存储，不要混用
- **幸存者偏差**：Tushare包含已退市股票，回测时必须主动纳入，否则结果虚高
- **数据校验**：下载后检查是否有缺失日期、异常价格（涨跌幅超过±20%但非涨跌停）

## 4. 股票池模块：Universe Filter

不要每天从5000多只股票里随机挑，系统必须先生成"可交易股票池"。

### 4.1 过滤条件

```text
1. 非ST、非*ST
2. 上市超过一年
3. 近20日平均成交额 > 5000万元
4. 当日未停牌
5. 当日非一字涨停或一字跌停（即开盘等于涨跌停价）
6. 非退市风险股
```

### 4.2 模块职责

回答一个问题：**哪些股票今天允许被系统考虑？**

很多策略不是死在买卖信号，而是死在垃圾股票、流动性、停牌、涨跌停和无法成交这些现实问题上。

## 5. 入场规则：Entry Rules

### 5.1 趋势条件

```text
1. 收盘价 > 60日均线
2. 20日均线向上（今日MA20 > 昨日MA20）
3. 大盘指数（沪深300）在20日均线之上
```

### 5.2 入场触发

```text
4. 收盘价突破过去20日最高价（突破信号）
5. 当日成交额 > 过去20日平均成交额（放量确认）
```

### 5.3 信号排序机制（新增，关键）

5个条件过滤全市场，在牛市中可能每天产生50-200个候选。必须有排序机制决定优先看哪些。

**推荐排序方式：相对强度评分**

```python
# 综合评分 = 相对强度分 + 成交量分 + 趋势强度分
relative_strength = 近20日涨幅 - 同期沪深300涨幅   # 相对强度
volume_ratio = 当日成交额 / 近20日平均成交额        # 量比
ma_slope = (MA20_today - MA20_20days_ago) / MA20_20days_ago  # 均线斜率

# 归一化后加权，取前10-15名作为当日候选池
```

**每日候选池上限：10-15只**，超出部分不考虑，等待下一个交易日重新评分。

### 5.4 第一版入场规则完整示例

```text
如果：
1. 收盘价 > 60日均线
2. 20日均线今日向上
3. 收盘价突破过去20日最高价
4. 当日成交额 > 过去20日平均成交额
5. 沪深300在20日均线之上

且 相对强度评分排名前15

则：
第二天开盘买入（不追涨超过2%）
```

不要一开始就追求复杂。复杂不等于有效。

## 6. 出场规则：Exit Rules

退出比买入更重要。

### 6.1 止损退出

```text
买入后收盘价亏损达到 -7%，次日开盘退出
```

### 6.2 趋势破坏退出

```text
收盘价跌破20日均线，次日开盘退出
```

### 6.3 止盈和利润保护

```text
盈利超过15%后，将止损价上移至成本价（保本止损）
盈利继续扩大时，跌破10日均线则减仓50%
最终止损：跌破20日均线清仓
```

### 6.4 第一版出场规则示例

```text
如果：
1. 买入后收盘价亏损 > 7%
或
2. 收盘价跌破20日均线
或
3. 盈利曾超过15%后又跌破10日均线

则：次日开盘卖出
```

## 7. 仓位规则：Position Sizing

仓位规则必须先于收益规则。

### 7.1 第一版仓位规则

```text
单票最大仓位：15%
单笔最大账户风险：1%
总仓位上限：60%
同行业持仓上限：30%
连续3笔亏损：新开仓仓位减半
单日亏损超过2%：当天不再开仓
系统最大回撤超过10%：暂停新开仓
```

### 7.2 仓位计算公式

```text
可承受亏损金额 = 账户总资金 × 单笔最大风险比例（1%）
持仓股数 = 可承受亏损金额 ÷ (入场价 - 止损价)
```

示例：

```text
账户资金：100,000元
单笔最大风险：1%  →  可承受亏损：1,000元

入场价：20元
止损价：18.6元（跌破20日均线）
每股风险：1.4元

最大可买股数：1,000 ÷ 1.4 ≈ 714股
A股按100股一手：买700股，金额约14,000元（仓位14%，符合单票上限15%）
```

## 8. 策略失效条件（新增，关键）

明确说明策略在哪些环境下不适用，避免盲目执行。

### 8.1 市场环境失效

```text
震荡市（沪深300连续20日涨跌幅在 ±5% 以内、方向不明）：
  → 趋势突破假信号频繁，手续费侵蚀严重，建议降仓或暂停

单边熊市（沪深300跌破年线且持续下行）：
  → 止损频繁触发，建议空仓等待
```

### 8.2 策略自身失效信号

```text
滚动3个月胜率 < 35%：触发策略审查
滚动3个月盈亏比 < 1.0：触发策略审查
连续5笔亏损：强制暂停，人工复盘后才能重启
```

### 8.3 验证重点回测年份

回测必须重点看这几个年份，不能只看牛市曲线：

| 年份 | 市场特征 |
|---|---|
| 2015下半年 | 单边暴跌 |
| 2016 | 熔断后震荡 |
| 2018 | 单边熊市 |
| 2019-2020 | 结构性牛市 |
| 2022 | 震荡下跌 |
| 2023-2024 | 低波动震荡 |

## 9. 回测系统：Backtesting

回测不是为了证明你能赚钱，而是为了发现系统会怎么死。

### 9.1 回测框架选择

**不建议**：
- `backtesting.py`：设计偏向单股票，多股票回测需要大量改造
- 第一版自写回测：2周内写出的回测框架容错率很低，bug难发现

**推荐**：
- `vectorbt`：性能好，多股票原生支持，适合日线级别批量回测
- `bt`：仓位管理逻辑更完善，API更友好
- 或自写，但至少预留4周而不是2周

选定框架后，**先用已知简单策略（如双均线）验证框架本身的正确性**，再跑目标策略。

### 9.2 必须统计的指标

```text
年化收益
最大回撤
夏普比率
胜率
盈亏比
单笔平均收益
最大连续亏损次数
持仓天数分布
换手率
手续费和滑点后的净收益
牛市、熊市、震荡市中的分段表现
```

### 9.3 A股回测必须处理的现实因素

```text
手续费：万分之2.5（买卖双向）
印花税：千分之1（仅卖出）
滑点：开盘价基础上加0.1%-0.3%模拟冲击成本
涨跌停无法成交：信号日涨停则跳过，不强制成交
停牌：信号日停牌则跳过
幸存者偏差：必须纳入已退市股票
未来函数：信号必须用T日收盘后的数据，T+1开盘成交
```

### 9.4 避免过拟合

不要试1000组参数找最优解。应该：

- 参数尽量少，规则有行为逻辑
- 分训练区间（2015-2020）和测试区间（2021-2024）
- 用不同市场阶段分段验证
- 做滚动窗口测试（rolling window）
- 看稳定性，而不是只看最高收益

## 10. 纸面交易、模拟交易和小资金实盘

回测通过后，不要马上全仓实盘。

### 10.1 第一层：信号观察（2-4周）

系统每天收盘后生成候选报告，只观察不交易，记录：
- 明日候选股票
- 买入理由和止损价
- 建议仓位

### 10.2 第二层：模拟交易（**4-6周，不要压缩**）

按信号模拟下单，记录：
- 信号是否合理
- 是否频繁误判
- 是否经常买不到或卖不出
- 涨跌停是否影响成交
- 数据延迟是否导致出错

**模拟盘时间至少4周，要覆盖至少一次完整的市场波动（涨-跌-震荡）。** 2周时间太短，很可能只看到单边行情，无法暴露策略在震荡市的失效问题。

### 10.3 第三层：小资金实盘

只用一小部分可承受损失的资金。第一阶段不是为了赚钱，而是验证：

> 系统在真实市场、真实滑点、真实情绪下还能不能执行。

## 11. 每日交易流程

### 11.1 盘前流程

```text
检查大盘环境（沪深300位置）
检查持仓风险（是否有止损触发风险）
确认候选股票是否仍满足条件
检查是否有异常公告、停牌或涨跌停风险
```

### 11.2 盘中流程

第一版不要盘中频繁决策，只处理：

```text
是否触发止损
是否无法成交
是否出现极端行情
是否出现系统异常
```

### 11.3 盘后流程

```text
更新行情数据
生成新信号（含排序评分）
记录交易日志
更新持仓收益
复盘系统表现
生成日报
```

### 11.4 每日系统报告模板

```text
【日期】
今日持仓（代码、成本、浮盈、止损价）
今日卖出信号
明日买入候选（Top 10，含评分）
风险暴露（总仓位、行业分布）
最大回撤（月度、年度）
策略状态：正常 / 降仓 / 暂停
异常日志
```

## 12. 软件架构设计

### 12.1 模块划分

```text
data_loader
  读取Tushare行情、复权因子、指数、行业、停牌、ST信息

data_cleaner
  清洗缺失值、复权（前复权统一存储）、处理涨跌停、处理停牌

universe_filter
  生成可交易股票池（每日）

signal_engine
  计算指标，生成买入/卖出信号

signal_ranker          ← 新增
  对买入候选按相对强度评分排序，输出Top N

risk_engine
  计算仓位、止损、总风险暴露、检查降仓/暂停条件

backtester
  历史回测，加入手续费、滑点、涨跌停限制、幸存者偏差

portfolio_manager
  管理持仓、现金、成交记录

report_generator
  每日生成交易报告

monitor
  监控异常、数据错误、仓位超限、程序错误

execution_engine       ← 后期
  接券商API或半自动下单
```

### 12.2 第一版不接券商API

第一版只需要做到：

```text
数据更新 → 信号生成 → 信号排序 → 仓位建议 → 风险检查 → 日报输出
```

## 15. 常见陷阱

### 15.1 一开始就做机器学习预测涨跌

不要上来就做LSTM、Transformer、强化学习或复杂因子模型。

金融市场噪音极大、非平稳、标签不稳定，A股日线数据量不足以支撑ML，非常容易过拟合。

第一阶段先跑通：

```text
规则系统 → 回测 → 模拟 → 小资金实盘 → 复盘
```

### 15.2 只看收益，不看回撤

一个策略年化收益很高但最大回撤50%，对个人来说未必能执行。真正要问的是：

> 这个系统亏损时，我是否还能坚持执行？

### 15.3 不停改规则

如果每亏一次就改参数，不是在优化系统，而是在追逐噪音。

建议：

```text
每周只复盘，不改系统
每月可以小改参数
每季度才能大改策略
所有修改必须记录原因和预期效果
```

### 15.4 过早自动下单

自动下单是最后一步，不是第一步。第一阶段让系统发信号，手动确认。

### 15.5 压缩模拟盘时间

模拟盘2周就进入实盘是常见错误。2周可能只覆盖单边行情，策略在震荡市的失效问题完全暴露不出来。

## 16. 90天执行计划（修订版）

### 第1周：前置准备（新增）

**数据和工具验证，不写业务代码。**

产出：
- 确认Tushare积分，验证全量历史数据可下载
- 运行数据验证脚本，确认停牌/ST/指数数据完整
- 搭建Python项目结构，配置Git
- 完成"交易系统说明书"初稿（边界、规则、仓位）

重点：

> 不要到第3周才发现数据下载不了。

---

### 第2周：完成系统说明书

产出：
- 交易系统说明书终稿
- 股票池规则文档
- 买卖规则文档
- 仓位规则文档
- 策略失效条件文档
- 交易日志模板

重点：

> 逻辑写清楚，不急着写代码。每个规则都要能回答"为什么这样设计"。

---

### 第3–4周：搭建数据层

产出：
- Tushare数据下载脚本（含断点续传）
- 全量A股日线历史数据库（含退市股）
- 前复权价格处理
- 停牌/ST/涨跌停标记数据
- 指数行情
- 数据完整性校验脚本

重点：

> 数据不干净，回测结果基本没意义。花足够时间做数据校验，而不是假设数据是对的。

---

### 第5–6周：做信号系统

产出：
- 股票池过滤模块（universe_filter）
- 指标计算模块（均线、成交额、突破）
- 买入/卖出信号生成模块
- 相对强度评分和排序模块（signal_ranker）
- 每日候选股票报告（可读的文本或简单HTML）

重点：

> 先检查信号是否符合常识。手动对比几只股票的历史走势和信号，确认没有未来函数。

---

### 第7–9周：做回测系统（调整为3周）

产出：
- 选定回测框架（vectorbt 或 bt），先用双均线验证框架正确性
- 集成信号系统和回测框架
- 手续费、印花税、滑点模拟
- 涨跌停无法成交处理
- 停牌处理
- 幸存者偏差纳入（使用含退市股的完整数据）
- 分段回测（分年份、分市场环境）
- 输出：最大回撤、胜率、盈亏比、夏普比率等

重点：

> 回测框架本身比策略更重要。框架错了，策略结果就是幻觉。先验证框架，再跑策略。

---

### 第10周：回测分析和参数确认

产出：
- 2015-2024年分段表现对比
- 参数稳定性分析（小幅调整参数，看结果是否剧烈变化）
- 样本外测试（用2022-2024年测试，用2015-2021年训练）
- 确认最终参数，冻结策略

重点：

> 这一周不是在优化参数，而是在确认策略是否足够稳定。不稳定的策略回到第5周重新设计规则。

---

### 第11–12周：模拟交易（开始，持续到第16周）

产出：
- 每日自动报告（信号、候选、当前持仓）
- 模拟持仓和模拟成交记录
- 错误日志
- 人工复盘记录

重点：

> 模拟交易至少持续4-6周（延伸到第90天之后）。要覆盖至少一次完整的市场涨跌周期，不要压缩。

---

### 第13周（第90天）：阶段复盘

产出：
- 90天系统回顾报告
- MVP Checklist完成情况
- 下一阶段计划（继续模拟 → 小资金实盘的时间表）

## 17. 推荐技术栈

```text
Python 3.10+
pandas / numpy
Tushare Pro（数据源主力）+ AKShare（备用/补充）
SQLite（小资金阶段）/ PostgreSQL（后期）
vectorbt 或 bt（回测框架）
matplotlib / plotly（可视化）
Streamlit（日报Dashboard）
Git（版本控制）
cron / GitHub Actions（定时更新数据和生成日报）
```

不要一开始就上：
- 微服务、Kubernetes、云部署
- 机器学习平台
- 自动下单API
- 高频行情系统

你现在最需要的是：

> 可理解、可调试、可复盘。

## 20. 最终判断标准

一个交易系统是否值得继续，看：

- 是否有清楚规则
- 是否能稳定复现
- 是否能控制亏损
- 是否能解释失败
- 是否能在不同市场阶段生存（特别是震荡市）
- 是否能被你长期执行

如果一个策略收益很高但你无法执行，它不是你的系统。

真正适合你的交易系统：

> 逻辑清楚、风险可控、复盘方便、执行压力不大、长期可迭代。
